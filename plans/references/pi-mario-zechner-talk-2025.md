This is the primary source behind [pi-principles-spec.md](../pi-principles-spec.md). The talk is a three-act argument:

- **Act 1 — Building Pi.** Why Claude Code and OpenCode failed Mario's needs (context invisibly mutated by the harness; tool definitions changing per release; system reminders injected at inopportune moments; tool output pruned past a token threshold; LSP errors injected into edit results; zero observability; zero model choice; shallow extensibility). How Pi responds: minimal core (4 tools — read/write/edit/bash, tiny system prompt), extension-first (TypeScript modules, hot-reloaded, distributed via NPM), self-modifying (the agent writes its own extensions from documentation Mario ships in the box), YOLO by default (no approval theatre), per-cwd JSONL sessions.
- **Act 2 — OSS in the age of clankers.** Slop PRs and slop issues drowning the tracker. Mitigations: auto-close unsolicited PRs with a request for a human-voice issue (added contributors get whitelisted), label and deprioritise OpenClaw-instance traffic, embed issues into 3D space to see clusters, declare "OSS vacation" by closing the tracker at will. Mitchell turned the auto-close pattern into [`vouch`](https://github.com/...).
- **Act 3 — Slow the fuck down.** Agents compound errors with no bottleneck and delayed pain. Review agents are an ouroboros. Models learn complexity from 90% garbage code on the internet; every agent decision is local; abstractions stack. "Sufficiently detailed spec" = a program — blanks get filled with internet-garbage defaults. Humans have bottlenecks and feel pain; agents don't. Use agents for boring/scoped/eval-checkable tasks; **read every line of critical code**.
